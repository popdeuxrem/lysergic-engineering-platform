# Reference Primitive Specification

## Normative Scope

**Artifact:** ECP-RP-v0.1.0-E0.2.2
**Parent:** ECP-RP-v0.1.0-E0.2 (Primitive Schema Foundation)
**Level:** L0 (Foundation Primitive)
**Dependencies:** identity

---

## 1. Purpose

Define the canonical representation of typed references between governed
objects. A reference connects a source identity to a target identity through
a declared semantic relationship type. Every reference is immutable after
creation and carries no execution semantics.

---

## 2. Constitutional Properties

### 2.1 Dual-Endpoint Identity

Every reference carries two identity endpoints:

- `source_id` / `source_uri` — Logical and canonical identity of the
  referencing object. Conforms to the Identity Primitive shape.

- `target_id` / `target_uri` — Logical and canonical identity of the
  referenced object. Conforms to the Identity Primitive shape.

Both endpoints are required and immutable after the reference is created.

### 2.2 Relationship Type

Every reference declares a `relationship` string that classifies the semantic
connection between source and target. Relationship values are registry
controlled — they must be registered in the relationship registry before use.

### 2.3 Immutability

Once created, a reference is immutable. No field may be mutated after
publication. A changed relationship between two objects requires a new
reference artifact, not mutation of an existing one.

### 2.4 Acyclic Dependency Constraint

References must not create cycles in the governed dependency graph. The
conformance checker enforces that the directed graph formed by references
remains acyclic.

### 2.5 No Execution Semantics

References describe static relationships between governed objects. They
must not encode execution logic, processing instructions, or backend-
specific routing. Execution systems consume references as data, not
as instructions.

---

## 3. JSON Schema

The normative schema is defined in `schemas/primitives/reference/reference.schema.json`
(Draft 2020-12).

**Canonical `$id`:** `https://specs.lysergic.dev/primitives/v1/reference.schema.json`

Reference does not use `$ref` to the Identity Primitive schema to maintain
L0 isolation. Instead, source and target fields mirror the identity shape
(`id` + `uri` pattern) with compatible type constraints.

---

## 4. Conformance

Reference conformance is defined by `conformance.yaml` (12 checks matching
the acceptance criteria). Key gates:

| Check | Requirement |
|-------|-------------|
| REF-001 | Source exposes immutable logical identity |
| REF-002 | Source exposes immutable canonical URI |
| REF-004 | Source fields conform to identity shape |
| REF-005 | Target fields conform to identity shape |
| REF-007 | No execution semantics present |
| REF-009 | No mutation after creation |
| REF-010 | Dependency graph remains acyclic |

---

## 5. Pydantic v2 Mapping Guidance

```python
from pydantic import BaseModel, Field
from typing import Literal

class Reference(BaseModel):
    source_id: str = Field(description="Logical identity of the source object")
    source_uri: str = Field(description="Canonical URI of the source object")
    relationship: str = Field(description="Registry-controlled relationship type")
    target_id: str = Field(description="Logical identity of the target object")
    target_uri: str = Field(description="Canonical URI of the target object")
```

All fields are required and immutable. Use a registry-backed enum or
validation function to constrain `relationship` to known types.

- **Version:** 0.1.0
- **Status:** READY_FOR_ACCEPTANCE

## 1. Purpose

This specification defines the foundational schema architecture for the Engineering Control Plane (ECP). It establishes the normative conventions governing how specifications, manifests, profiles, and evidence are structured, identified, validated, and evolved across the LEP program.

## 2. Normative Precedence

The following order of authority applies:

1. **Constitutional Rules** — LEPRA-GM-v1.0 governance model (highest)
2. **Evolution Rules** — LEPRA-ER-v1.0 change management
3. **Normative Specifications** — ECP artifact specifications
4. **Governance Manifests** — Artifact-specific manifest files
5. **Conformance Profiles** — Validation and conformance definitions

## 3. Repository Structure

The repository organizes ECP artifacts under the following top-level directories:

```
docs/specifications/     — Normative specifications (one per artifact)
manifests/               — Governance manifest files
profiles/                — Conformance profiles
evidence/                — Acceptance and evidence packages
```

## 4. Namespace Policy

All ECP artifacts use the following identity scheme:

```
ECP-RP-v{semver}-E{major}.{minor}
```

Where:
- `RP` denotes the repository platform track
- `v{semver}` is the program version
- `E{major}.{minor}` is the ECP artifact number

## 5. Versioning Policy

- Artifact versions follow semantic versioning (MAJOR.MINOR.PATCH)
- Specification versions are immutable after acceptance
- Amended specifications receive a new minor version
- Breaking changes require a new major version

## 6. Reference Strategy

- Specifications may reference other ECP specifications by artifact ID
- External references use URI notation with the `https://specs.lysergic.dev/` base
- References must include the target artifact version

## 7. Dependency Graph

```
E0.1 — Schema Architecture Foundation
  └── E0.2 — Primitive Schema Foundation (planned)
       └── E0.3 — Audit & Validation Schema (planned)
```

## 8. Primitive Order

The following primitive types are defined at the E0.1 level:
- `ArtifactId` — unique artifact identifier
- `ArtifactState` — lifecycle state identifier
- `ValidationResult` — PASS / FAIL / PENDING
- `GovernanceDecision` — ACCEPTED / REJECTED / PENDING

## 9. Validation Architecture

- Each artifact defines a conformance profile
- Validation executes against the conformance profile
- Results are recorded in the evidence package
- Acceptance requires all validation gates to pass

## 10. Evolution Rules

- Artifacts may be amended only through the Evolution Request process
- Evolution requests are evaluated against LEPRA-ER-v1.0
- Breaking changes require Portfolio Authority approval

## Implementation Summary

### Request Context Middleware

Created `src/observability/middleware.py` — FastAPI middleware that:

- Generates `request_id` per incoming request
- Accepts `X-Correlation-ID` header when provided
- Generates `correlation_id` when header absent
- Attaches both identifiers to `request.state`
- Logs lifecycle events: `request.started`, `request.completed`, `request.failed`
- Sets `X-Request-ID` and `X-Correlation-ID` response headers
- Captures method, path, status_code, and duration_ms in structured logs

### Context Model

Created `src/observability/context.py` — `RequestContext` dataclass with `request_id` and `correlation_id`.

### Structured Logging Expansion

Updated `src/observability/configure_logging.py` — `JsonFormatter` now includes optional request fields (`request_id`, `correlation_id`, `method`, `path`, `status_code`, `duration_ms`) when present in the log record.

### API Integration

Updated `src/application/app.py`:

- Replaced inline `set_request_id` middleware with `request_context_middleware`
- Removed `uuid4` import (now handled by observability module)
- All existing endpoints preserved: `GET /`, `GET /health`, `GET /version`, `GET /api/v1/health`, `GET /api/v1/version`

### Error Context Integration

- Existing `build_error_response` in `src/api/exceptions.py` uses `request.state.request_id` with fallback
- M1.3 error envelope fully preserved: `{"error": {"code", "message", "request_id"}}`

### Dependency Boundary Protection

- Middleware imports from `src/observability.context` only
- No FastAPI middleware → Domain or SQLAlchemy paths
- Logging remains in `src/observability` package with no business logic
